1. Problem Statement 
India hosts one of the largest and most complex competitive examination ecosystems in the 
world. Millions of aspirants appear each year for examinations conducted by UPSC, SSC, 
IBPS, RBI, and state -level Public Service Commissions, each carrying distinct e ligibility 
criteria covering age limits, educational qualifications, social category reservations, 
permissible attempt counts, and domicile requirements. 
 
Despite this scale, aspirants currently manage eligibility tracking through a fragmented 
combination of physical marksheets, handwritten notes, bookmarked government websites, 
and social media groups. No unified platform exists that: 
● Automatically reads and stores academic credentials from physical documents using 
on-device OCR. 
● Applies exact eligibility rules of multiple competitive examinations to a candidate’s 
profile in real time. 
● Tracks the number of attempts consumed across examinations from a single interface. 
● Alerts the candidate to approaching registration deadlines and provides a direct 
pathway to official portals. 
● Projects future eligibility windows based on the candidate’s age, category, and 
academic standing. 
● sometimes aspirant fill to keep in mind the number of attempts they are left with  
 
The consequence is twofold: aspirants miss registration deadlines due to poor tracking, and 
they invest preparation time in examinations for which they are ineligible, discovering this 
only at the point of application rejection. Yogya is designed to elimin ate both failure modes 
through intelligent automation. 
2. Objectives 
2.1 Primary Objectives 
● Develop a cross -platform mobile application (Android and iOS) using the Flutter 
framework that serves as an intelligent exam eligibility and attempt -tracking 
companion for Indian competitive examination aspirants. 
● Implement an on-device OCR pipeline capable of extracting structured academic data 
from photographs of 10th Marksheets, 12th Marksheets, and Graduation Degree / 
CGPA Certificates with a confidence score of 95 % or above. 
● Build a rule-based Eligibility Engine that evaluates a candidate’s profile against the 
exact eligibility criteria of multiple major Indian competitive examinations 
simultaneously and produces a real-time match percentage per exam. 
2 
 
● Design a personalised Eligibility Timeline that projects the candidate’s valid 
examination windows chronologically, identifies the nearest deadline, and provides a 
direct link to the official registration portal. 
● Provide a secure, privacy -preserving architecture that stores only structured 
extracted data, purges raw document images immediately after OCR extraction, and 
encrypts all credentials using hardware-backed storage. 
 
2.2 Secondary Objectives 
● Surface a contextual ‘Next Steps’ card on eligibility confirmation listing additional 
documents required by each specific examination with links to the official information 
source. 
● Design a comprehensive Profile, Settings, and Help & Support interface allowing 
aspirants to manage their academic identity, social category, notification preferences, 
and security settings from one place. 
● Establish an extensible Clean Architecture that allows new examinations, new 
eligibility rules, and new document types to be added without rebuilding the core 
engine. 
● Deliver a brand-consistent user interface with smooth animations, a persistent bottom 
navigation paradigm, and a hamburger -triggered side drawer for secondary 
navigation.

6. App Flow / User Journey 
6.1 First-Time User Journey 
● User downloads and launches Yogya. Splash Screen checks for a JWT token in secure 
storage. None found - routes to Login Screen. 
● User registers via email/password or Google OAuth. JWT issued and stored in the 
hardware-backed keystore. 
● Profile Screen opens: user enters Date of Birth, Gender, Social Category, State of 
Domicile, Educational Status, and Primary Exam Goal. 
● A ‘Complete Your Profile’ nudge directs the user to the Documents screen to upload 
their 10th Marksheet (first compulsory document). 
● User opens the OCR viewfinder, aligns the marksheet within the frame. The 4 -layer 
pipeline processes the image in real time, showing a progress bar from 0  % to 100 % 
with step badges. 
7 
 
● Extracted data is shown for user verification. On confirmation, the raw image is 
purged from the device. The Eligibility Engine performs a first pass using the now -
verified date of birth. 
● User uploads 12th Marksheet and Graduation Certificate through the same OCR flow. 
With all three core documents verified, the Live Eligibility Pulse shows full match 
percentages and the Dashboard surfaces the first eligible exam’s registration 
deadline. 
 
6.2 Returning User Journey (Daily Use) 
● Splash Screen detects a valid JWT - routes directly to Dashboard. Cold-start under 2 
seconds. 
● Dashboard shows current UPSC attempt count (e.g., 3  / 6), nearest deadline with 
countdown (e.g., ‘CDS II 2024 Application - 12 Days Remaining’), and recent document 
verification card. 
● User opens the Eligibility tab, views the Live Eligibility Pulse, and updates CGPA 
after semester results. The engine recomputes all exam matches in real time. 
● User opens the Timeline tab. Sees UPSC CSE Prelims (ACTIVE, Eligible, Attempt 
#2 / 6), AFCAT (Ineligible - age exceeded by 2 months), and CDS II 2024 (Eligible with 
‘Register Now’ button). 
● User taps ‘Register Now’ on CDS  II. An ‘Eligible - Next Steps ’ card opens listing 
required documents (12th certificate, degree certificate, NCC C -certificate if 
applicable, medical fitness clearance) with links to the official UPSC portal.

8.3 OCR Pipeline - 4 Layers 
● Layer 1 - Image Pre -processing: Perspective correction, deskewing, contrast 
enhancement, and noise reduction using OpenCV bindings or Flutter’s image 
package. 
● Layer 2 - Layout Detection: Identifies document type (marksheet vs. degree), detects 
table regions and column headers using a lightweight on-device TFLite model. 
● Layer 3 - Text Extraction: Runs Google ML Kit Text Recognition on detected regions 
to extract raw text strings. 
● Layer 4 - Structured Data Parsing: A rule-based parser maps text to a typed schema 
(board, year, marks, aggregate  %, DOB). Per-field confidence scores calculated. Raw 
image purged via dart:io File.delete() on success.

9.2 Data Flow 
● User scans document → OCR pipeline extracts structured fields → 
AcademicDocument record written to local DB. 
● AcademicDocument write triggers EligibilityNotifier → EligibilityEngine reads 
UserProfile + all AcademicDocuments → computes new EligibilityResult per 
TrackedExam. 
● Dashboard and Timeline observe EligibilityResult stream via Riverpod 
FutureProvider → only the changed card rebuilds. 
● On an app  open with an active  session, TrackedExam and DeadlineAlert data is 
refreshed from the remote API using a stale-while-revalidate caching strategy. 
 
9.3 Privacy & Data Minimisation 
16 
 
● Raw document images retained only in volatile memory (RAM) during OCR 
processing. Purged immediately on extraction completion. 
● Social category stored as an encrypted string in UserProfile. No physical certificate 
document is ever stored or transmitted. 
● JWT tokens stored exclusively in flutter_secure_storage (iOS Keychain  / Android 
Keystore). Never in SharedPreferences or any plaintext file. 
● Only structured EligibilityResult and UserProfile fields are synced to the backend. 
Document images and raw OCR text are never transmitted.
